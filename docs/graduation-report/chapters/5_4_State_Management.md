# 5.4 إدارة حالة التطبيق (State Management)

تعتمد **منصة الحكومة الإلكترونية** على مكتبة **Zustand** لإدارة الحالة البرمجية في الواجهة الأمامية. تم اختيار هذه المكتبة لكونها توفر أداءً عالياً مع سهولة في التعامل مقارنة ببدائل أخرى مثل Redux، وهو ما يتناسب مع متطلبات الأنظمة اللحظية.

## 1. فلسفة إدارة الحالة
تم تقسيم الحالة البرمجية إلى "متاجر" (Stores) مستقلة وظيفياً، مما يسمح بفصل الاهتمامات (Separation of Concerns) وسهولة تتبع تدفق البيانات:

*   **متجر التوثيق (Auth Store):** مسؤول عن حفظ بيانات المستخدم ورمز الـ JWT. يتميز هذا المتجر بخاصية **الديمومة (Persistence)**، حيث يتم حفظ الحالة في `localStorage` لضمان عدم فقدان تسجيل الدخول عند تحديث الصفحة.
*   **متجر التنبيهات (Notification Store):** يدير قائمة الإشعارات الواردة للمواطن، ويقوم بحساب عدد الرسائل غير المقروءة وتحديث الحالة محلياً فور قراءة التنبيه.
*   **متجر التواصل اللحظي (Socket Store):** مسؤول عن دورة حياة اتصال الـ WebSocket، حيث يقوم بفتح الاتصال فور تسجيل الدخول وإغلاقه عند الخروج، مما يضمن كفاءة استهلاك موارد الخادم.
*   **متجر المرور (Traffic Store):** يدير حالة "عروض الشراء" النشطة، ويتحكم في ظهور النوافذ المنبثقة (Pop-ups) لحظياً عند تلقي عرض جديد من بائع آخر.

## 2. التكامل مع الواجهات (UI Integration)
يتم ربط هذه المتاجر مع مكونات React بشكل مباشر، حيث تؤدي أي عملية تحديث في المتجر (مثل تلقي إشعار جديد عبر Socket) إلى إعادة صيرورة (Re-render) الأجزاء المعنية فقط من الواجهة، مما يوفر تجربة مستخدم سلسة واستجابة سريعة جداً.

## 3. مزامنة البيانات مع الخادم
تعمل متاجر Zustand كجسر بين واجهات البرمجة (REST API) والواجهة المستخدمة:
1. يتم استدعاء الـ API لجلب البيانات.
2. يتم تحديث المتجر بالبيانات الجديدة.
3. تنعكس التغييرات تلقائياً على كافة الصفحات التي تستخدم هذا المتجر.

يضمن هذا الأسلوب المعماري أن تظل حالة التطبيق متسقة دائماً مع البيانات الحقيقية في قاعدة البيانات، مع توفير سرعة استجابة فائقة في التفاعلات اللحظية التي تعد جوهر مشروع منصة الحكومة الإلكترونية.
